### 2.2 预警触发引擎 (AlertTriggerServiceImpl) — 核心逻辑

#### 2.2.1 实时触发 — checkOnPosting

触发时机：每次成本入账成功后调用。遍历所有启用规则，按范围匹配、计算当前值、比较阈值、检查静默。

| 规则类型 | 计算公式 | 结论 |
|---------|---------|------|
| EXEC_RATE | `actualCost × 100 / totalBudget`，保留2位小数 | ✅ 正确处理除零（返回0） |
| SINGLE_AMOUNT | 入账记账的 `amount` | ✅ 直接取用 |
| BALANCE_RATE | `budgetBalance × 100 / totalBudget`，保留2位小数 | ✅ 正确处理除零 |

| 测试用例 | 预期结果 | 实际结果 | 结论 |
|---------|---------|---------|------|
| 全部预算=0时执行率计算 | 返回0，不触发预警 | `totalBudget.compareTo(ZERO) <= 0` 时返回0 | ✅ |
| 单笔入账金额触发阈值 | 创建预警记录，写入日志 | 按预期执行 | ✅ |
| 静默期内重复触发 | `countRecentAlert > 0` 跳过 | 按预期执行 | ✅ |
| scopeType=0(全局) | 对所有项目生效 | `matchesScope` 返回 true | ✅ |
| scopeType=1(指定项目) | 仅匹配 `scopeValue` 中列出的项目ID | `containsId` 逗号分隔匹配 | ✅ |
| scopeType=2(指定科目) | 仅匹配 `scopeValue` 中列出的科目ID | `containsId` 逗号分隔匹配 | ✅ |
| 通知开关关闭 | 仅记录 TRIGGER 日志，不记录 NOTIFY | 当 `notifyEnabled != "1"` 时跳过 | ✅ |
| 通知开关开启 | 记录 TRIGGER + NOTIFY 两条日志 | 正确写入两条 | ✅ |

#### 2.2.2 定时扫描 — checkDailyRules

触发时机：每日定时任务。仅处理 `OVERDUE`（项目逾期）和 `INACTIVE`（长期无入账）两类规则。

## 九、修改建议（架构层面）

### 9.1 预警触发解耦

当前 `checkOnPosting` 与入账在同一事务中（P1），建议：

```java
// CostApprovalServiceImpl.postCost 尾部
try {
    alertTriggerService.checkOnPosting(record.getProjId(), record);
} catch (Exception e) {
    log.warn("预警检查失败: projId={}, billId={}", record.getProjId(), record.getBillId(), e);
    // 不阻塞入账
}
```

### 9.2 定时扫描增加项目状态过滤

```java
// checkDailyRules 中
List<ProjInfo> projects = projInfoMapper.selectProjInfoList(new ProjInfo());
// 增加过滤：只保留已立项、进行中、已完工的项目
projects = projects.stream()
    .filter(p -> Arrays.asList("2", "4", "5").contains(p.getStatus()))
    .collect(Collectors.toList());
```

### 9.3 操作幂等性保护

`ProjAlertRecordServiceImpl` 各操作方法增加状态前值校验，防止重复操作。

### 9.4 前端统一错误处理

为 alertDashboard、alertRecord、alertRule 的所有 API 调用添加 `.catch()` 错误提示。

## 十、测试结论

预警监控模块在**架构设计**上合理，5种预警类型覆盖了成本管理的主要风险场景（预算执行率、大额入账、余额不足、项目逾期、长期无活动）。实时触发与定时扫描的双轨机制设计良好。

**主要风险点**：
1. **P0：逻辑错误** — OVERDUE/INACTIVE 规则对非进行中项目也会计算预警（草稿项目逾期、草稿项目无入账），这是业务逻辑错误
2. **P1：架构耦合** — 预警异常会回滚入账事务，监控功能不应阻塞核心业务
3. **P1：性能与逻辑** — 定时扫描无项目状态过滤，拉取全部项目做双重循环
4. **P2：前端缺陷** — 多处缺少错误处理和防重复提交保护
5. **P3：技术债务** — 13 项低优先级改进建议

**修复优先级建议**：先修 P0（逻辑错误）和 P1（解耦+过滤）→ P2（前端错误处理+防重）→ P3 跟进。
