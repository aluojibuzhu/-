## 三、后端逻辑测试 (边缘情况)

### 3.1 状态机完整性

#### 项目立项状态流转图
```
DRAFT(0) ──提交──→ PENDING(1) ──通过──→ APPROVED(2) ──手动推进──→ IN_PROGRESS(4) ──→ COMPLETED(5)
   │                   │                    │
   └── 驳回 ←──────────┘                    │
   REJECTED(3) ←────────────────────────────┘
      │
      └── 重新编辑(回到DRAFT)，或删除
```

| 检查项 | 结论 |
|--------|------|
| 所有状态转换都有 `ProjectStatus.require` 校验 | ✅ |
| 删除仅允许 DRAFT/REJECTED | ✅ |
| 驳回仅允许 PENDING | ✅ |
| 提交仅允许 DRAFT/REJECTED | ✅ |
| COMPLETED 状态定义在枚举中但无任何代码引用 | ⚠️ 状态未完全实现 |

#### 工时/报销状态流转图
```
DRAFT(0) ──提交──→ PENDING(1) ──通过──→ APPROVED(2) ──入账──→ POSTED(4)
   │                   │                    │
   └── 驳回 ←──────────┘                    │
   REJECTED(3) ←────────────────────────────┘
      │
      └── 重新编辑(回到DRAFT)，或删除
```

| 检查项 | 结论 |
|--------|------|
| POSTED 状态不可逆 | ✅ |
| 已入账不可删除 | ✅ |
| 审批通过后可单独入账(postApproved) | ✅ |
| 审批通过入账可一步完成(approveAndPost) | ✅ |

### 3.2 并发安全性

| 场景 | 风险评估 | 结论 |
|------|---------|------|
| 同时提交同一草稿 | 两次都通过 require 校验，第二次更新覆盖第一次 | ⚠️ P2 |
| 同时审批同一单据 | **P0 竞态条件** (见 B-10) | ❌ |
| 同时入账同一单据 | 唯一约束 `uk_cost_posting_bill` 阻止重复 | ✅ |
| 序列号生成并发 | MySQL AUTO_INCREMENT 保证唯一 | ✅ |
| 项目保存时WBS删除+重插 | `@Transactional` 保证原子性 | ✅ |

### 3.3 数据一致性

| 检查项 | 结论 |
|--------|------|
| `totalBudget` = SUM(`nodeBudget`) | ✅ saveDraft 中计算并更新 |
| `budgetBalance` = `totalBudget` - `actualCost` | ✅ 每次入账时重算 |
| `actualCost(proj_info)` = SUM(`actualCost(wbs_node)`) | ⚠️ P2 — 分别更新两张表，若WBS节点更新失败但项目更新成功（理论上在同一事务中不会发生，但缺少跨表校验） |
| `actualCost(proj_cost_allocation)` = SUM(入账记录) | ⚠️ P3 — 无定期对账机制 |

### 3.4 安全校验

| 检查项 | 结论 |
|--------|------|
| 所有Controller方法都有 `@PreAuthorize` 权限注解 | ✅ |
| 后端校验了项目状态才能操作工时/报销 | ✅ |
| SQL 使用参数化查询，无注入风险 | ✅ |

## 八、修改建议（总体架构层面）

### 8.1 增加乐观锁

对所有涉及状态变更的 UPDATE 操作，在 SQL WHERE 子句中增加旧状态条件：

```xml
<!-- 示例：WorkHourMapper.xml -->
<update id="updateWorkHourStatus">
    update wh_work_hour 
    set status = #{newStatus}, update_by = #{updateBy}, update_time = now()
    where wh_id = #{whId} and status = #{expectedStatus}
</update>
```

然后在 Service 层判断 `updateCount == 0` 时抛出并发冲突异常。

### 8.2 统一审批审计字段

建议为 `proj_info` 表增加 `approve_by`（审批人）和 `approve_time`（审批时间）字段，与工时单、报销单对齐。

### 8.3 WBS/分配表改为逻辑删除

`proj_wbs_node` 和 `proj_cost_allocation` 应使用 `del_flag` 逻辑删除，保留历史数据用于审计追踪。

### 8.4 增加成本入账前置校验

在 `postCost` 中，对 `proj_cost_allocation` 更新前先检查 `affected rows`，若为0则记录警告日志或抛出异常。

### 8.5 成本对账机制

建议增加定时任务或手动触发的成本对账功能，校验：
- `proj_info.actual_cost` = SUM(`proj_wbs_node.actual_cost`)
- `proj_wbs_node.actual_cost` = SUM(`proj_cost_allocation.actual_cost`)
- `proj_cost_allocation.actual_cost` = SUM(`cost_posting_record.amount`)

## 九、测试结论

系统在**功能完整性**方面表现良好，8个业务模块的核心 CRUD 和状态流转逻辑均正确实现。后端校验较为完善，金额计算使用 BigDecimal 保证了精度。

**主要风险点**：
1. **并发安全**（P0）：缺少乐观锁机制，高并发下存在状态跳跃和数据不一致风险
2. **审计追溯**（P1）：项目审批缺少审批人/审批时间字段，WBS/分配表物理删除导致历史数据不可追溯
3. **前端防护**（P2）：部分审批操作按钮缺少防重复提交的 loading 保护
4. **技术债务**（P3）：共15项低优先级改进建议，可在后续迭代中逐步解决

**建议修复顺序**：先修复 P0（并发安全）→ P1（审计字段 + 校验完善）→ P2（前端防护 + 数据库）→ P3 逐步跟进。
