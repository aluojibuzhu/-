## 三、发现的 BUG

### P0 — 阻塞上线

| ID | 模块 | 描述 |
|----|------|------|
| **B-23** | 审批入账 | `PUT /cost/approval/approve/PROJECT_ESTABLISHMENT/{id}` 报错"不支持的单据类型"。项目立项无法通过统一审批入口审批，只能用 `PUT /project/info/approve/{id}` 单独审批 |

### P1 — 重要

| ID | 模块 | 描述 |
|----|------|------|
| **B-12** | 成本入账 | 项目创建时的成本分配记录使用一级科目ID(1-7)，但工时/报销使用二级科目ID(103,201等)，导致 `increaseActualCost` 匹配不到分配记录 → 入账失败报"成本科目预算分配记录不存在"。当前绕过方案是在项目创建时额外添加二级科目分配记录 |
| **B-26** | 成本入账 | `budgetBalance` 计算持续异常。实测数据：totalBudget=100,000，actualCost=100,800 时 budgetBalance=-10,800（预期 -800）；totalBudget=100,000，actualCost=105,400 时 budgetBalance=-25,400（预期 -5,400）。偏差值不固定 |
| **B-28** | 项目立项 | **客户必填但名称不显示**：`validateDraft` 校验 `customerId != null` → 必须选客户，但 `saveDraft` 从未查询客户表回填 `customerName`。而 `WorkHourServiceImpl` 和 `ReimbursementServiceImpl` 的 `fillAndValidate` 都会从 DB 查回填关联名称。结果：项目列表中所有项目的"客户名称"列显示为空 |
| **A-05** | 预警引擎 | OVERDUE 规则对草稿/驳回/审批中的项目也计算逾期天数（只排除了 status='5'） |
| **A-06** | 预警引擎 | INACTIVE 规则对草稿/驳回项目也计算"无入账"天数 |

### P2 — 中等

| ID | 模块 | 描述 |
|----|------|------|
| **B-27** | 成本入账 | 一级科目 allocation 的 `actualCost` 始终为0，只有二级科目 allocation 被更新。缺少从二级到一级的金额汇总机制 |
| **A-07** | 预警引擎 | `checkDailyRules` 加载全部项目无状态过滤 |

### P3 — 低优先

| ID | 模块 | 描述 |
|----|------|------|
| **B-25** | 项目立项 | allocations 中 `nodeId=null` 导致 SQL error "Column 'node_id' cannot be null" |
| — | URI编码 | 旧测试数据中部分项目名称有中文乱码（已通过 heredoc + UTF-8 头解决） |
