# MVP 主流程闭环 — 全链路测试用例

> 编写日期：2026-05-09  
> 测试范围：PRD_MVP.md 3.1节「主流程（MVP闭环）」→ 项目立项 → 预算编制 → 成本填报 → 审批入账 → 预警监控 → 成本看板  
> 数据注入方式：**仅通过"新建项目"功能模块注入测试数据**（`POST /project/info/draft`）  
> 前置条件：系统中已存在至少一个客户和成本科目体系（种子数据或手工创建）

---

## 前置条件

| 项目 | 值 | 获取方式 |
|------|-----|---------|
| 客户 | `customerId = 1`, `customerName = "测试建筑甲方公司"` | 种子数据或 `POST /project/customer` |
| 一级成本科目(人工费) | `categoryId = 1`, `categoryName = "人工费"`, `unitPrice = 500.00` | 种子数据或 `POST /system/costCategory` |
| 二级成本科目(劳务费) | `categoryId = 2`, `categoryName = "劳务费"`, `parentId = 1` | 同上 |
| 二级成本科目(材料费) | `categoryId = 3`, `categoryName = "材料费"`, `parentId = 4` | 同上 |
| 测试用户(项目经理) | `username = "pm"`, 具有 `project:projInfo:add/submit` 权限 | 若依用户管理 |
| 测试用户(管理员) | `username = "admin"`, 具有 `project:projInfo:approve` 等审批权限 | 若依用户管理 |
| 测试用户(项目成员) | `username = "member"`, 具有 `cost:workHour:add` / `cost:reimbursement:add` | 若依用户管理 |

> **注入原则**：以下所有测试用例的**新建项目**数据均通过 `POST /project/info/draft` 注入，包含完整的 `projInfo`、`wbsNodes`、`allocations` 三部分。项目编号由系统自动生成（PRJ-YYYYMMDD-NNN）。

## 一、阶段一：项目立项与预算编制（PRD 3.2 阶段一/二）

### TC-1.1 标准项目立项 — 完整方案保存草稿

| 项目 | 内容 |
|------|------|
| **用例编号** | TC-1.1 |
| **用例名称** | 标准项目立项：完整 WBS 节点 + 成本科目预算 → 保存草稿 |
| **测试目的** | 验证 `POST /project/info/draft` 接受完整的立项方案并正确生成编号、节点、分配记录 |
| **前置条件** | 客户ID=1 存在，科目ID=1,2,3 存在 |
| **注入方式** | `POST /project/info/draft`（新建项目模块） |

**注入数据（Request Body）**：
```json
{
  "projInfo": {
    "projName": "测试项目-住宅小区一期工程",
    "customerId": 1,
    "planStartDate": "2026-06-01",
    "planEndDate": "2026-12-31",
    "contractAmount": 5000000.00,
    "projDesc": "MVP闭环测试用项目：某住宅小区一期主体及配套工程"
  },
  "wbsNodes": [
    { "nodeName": "基础工程", "planFinishDate": "2026-07-15", "nodeBudget": 800000.00 },
    { "nodeName": "主体结构", "planFinishDate": "2026-10-15", "nodeBudget": 2500000.00 },
    { "nodeName": "装饰装修", "planFinishDate": "2026-12-15", "nodeBudget": 1200000.00 },
    { "nodeName": "竣工交付", "planFinishDate": "2026-12-31", "nodeBudget": 500000.00 }
  ],
  "allocations": [
    { "nodeId": null, "categoryId": 2, "allocationAmount": 400000.00 },
    { "nodeId": null, "categoryId": 3, "allocationAmount": 400000.00 }
  ]
}
```

> **说明**：`allocations` 中的 `nodeId: null` 表示新建场景下暂不指定节点（后续由服务端根据插入顺序映射）。实际测试中，若前端表单已支持节点-科目矩阵，此处应填具体节点ID。

**预期结果**：
| 验证点 | 预期值 |
|--------|--------|
| HTTP 状态码 | 200 |
| 响应 `data.projId` | 新生成的ID |
| 响应 `data.projNo` | 格式 `PRJ-YYYYMMDD-NNN` |
| 响应 `data.status` | `"0"`（草稿） |
| 响应 `data.totalBudget` | `5000000.00`（WBS节点预算之和） |
| `proj_info` 表记录 | 1条，`status='0'`, `customer_name` 已回填 |
| `proj_wbs_node` 表记录 | 4条，`node_no` 分别为 `ND-001` ~ `ND-004` |
| `proj_cost_allocation` 表记录 | 2条，`allocation_amount` 与注入一致 |

## 八、完整端到端测试脚本流程

### E2E-1：标准 MVP 闭环（项目A）

```
第一步 — 项目立项
  POST /project/info/draft          → 创建项目A，得到 projId
  POST /project/info/submit/{projId} → 提交审批

第二步 — 立项审批
  PUT /cost/approval/approve/PROJECT_ESTABLISHMENT/{projId} → 审批通过

第三步 — 手动将项目推进到进行中（如需）
  直接更新 status='4'（或通过其他接口）

第四步 — 填报工时
  POST /cost/workHour/draft         → 工时1（8h，施工，amount=4000）
  POST /cost/workHour/submit/{whId} → 提交审批

第五步 — 填报报销
  POST /cost/reimbursement/draft    → 报销1（15000，材料费）
  POST /cost/reimbursement/submit/{reimburseId} → 提交审批

第六步 — 统一审批入账
  PUT /cost/approval/approve/WORK_HOUR/{whId}     → 工时入账
  PUT /cost/approval/approve/REIMBURSEMENT/{reimburseId} → 报销入账

第七步 — 验证预警
  GET /alert/record/list?projId={projId}   → 检查是否生成预警记录
  GET /alert/dashboard/summary             → 检查汇总统计

第八步 — 处理预警
  PUT /alert/record/confirm/{alertId}      → 确认预警
  PUT /alert/record/close/{alertId}        → 关闭预警

第九步 — 验证成本累积
  GET /project/info/{projId}               → actualCost=19000, budgetBalance=4981000

第十步 — 验证看板
  GET /alert/dashboard/budgetTrend?projId={projId}&periodType=month
  GET /alert/dashboard/categoryCompare?projId={projId}
```
